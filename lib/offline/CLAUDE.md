# CLAUDE.md — lib/offline (client offline layer)

## Learnings (agent-maintained)
- The `ds-offline` IndexedDB is SHARED across modules (stores `wal` + `packs`). NEVER call `indexedDB.open` with a module-local pinned version — once any module upgrades the db (e.g. v1→v2 to add a store), a stale pinned open throws `VersionError` and that module's feature-detect fallback silently disables it. All opens go through `lib/offline/idb.ts` (`openOfflineDb`/`withStore(storeName, …)`); adding a store = bump `DB_VERSION` there and create it in the ONE `onupgradeneeded` (which must create ALL stores — first-open order between modules is arbitrary).
- Programmatic `fetch()` sends `Accept: */*`, so a `/api/q-image/<key>?w=540` fetch misses the avif/webp negotiation an `<img>` gets and downloads the full-size original. Pass `headers: { accept: "image/avif,image/webp,image/*,*/*;q=0.8" }` explicitly (see `downloadPack` in packs.ts) — otherwise stored bytes exceed the server's `estimatedImageBytes` (which stats the 540 avif variant first). The route sends no `Vary`, so Cache Storage put/match/delete key purely on URL.
- Cache Storage (`caches`) is SECURE-CONTEXT-gated exactly like service workers (IndexedDB is NOT): on the plain-http LAN audit origin (`http://100.110.64.90:3100`) `typeof caches === "undefined"`, so any pack/cache feature correctly no-ops as "unsupported" there. Browser-smoke the FULL download/cache flow on `http://localhost:3100` (loopback = secure context); the LAN origin can only assert affordance presence/dialog copy. UI copy should distinguish this permanent `unsupported` from a retryable network failure — «спробуйте ще раз» on an insecure origin is a lie.
- A Serwist runtime `CacheFirst({ cacheName: "q-images" })` lane only ever reads ITS OWN named cache — it never consults the app-owned `ds-pack-images` pack cache, so a plain `<img src="/api/q-image/…?w=540">` on an offline page misses the stored pack bytes and errors once the network is dead. Serve pack images from the page via `caches.match(url)` (searches EVERY cache, ds-pack-images included) → blob → `URL.createObjectURL` (revoke on cleanup), falling back to the network URL on miss (see `PackImage` in app/offline-practice/runner.tsx). Applies to any future app-owned cache the SW doesn't route.
- packs.ts write-order invariant: images FIRST, IndexedDB row LAST — and ALL cache deletions (deletePack + downloadPack's overwrite pruning) run only AFTER the row mutation succeeds, filtered through a keep-set of image keys every OTHER stored pack references (scopes overlap: mistakes/saved share images with topic packs). Any future pack-content mutation must keep both halves or an interrupted write strands a LISTED pack whose images are gone (the one state the module promises can't happen); orphaned UNLISTED images are the accepted harmless direction.
