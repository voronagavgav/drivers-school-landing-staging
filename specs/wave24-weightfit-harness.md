# SIGNAL — Wave 24: per-user FSRS weight-fit harness (build now, apply data-gated)

Source: `docs/research/BEYOND-CURRENT-RESEARCH-2026-07-13.md` finding #1 (HIGH confidence, top-3):
optimized FSRS weights beat defaults by ~0.019 absolute log-loss (RMSE ~28% relative) on the
srs-benchmark; Anki 24.06+ has NO minimum review count; defaults are medians of 20k collections so
~half of users sit off-target. OPEN question the survey couldn't close: fitted-vs-default quality
specifically at 200–2000 reviews / 2–8 weeks — so the harness must MEASURE that on our own data
before anything is applied. REFUTED (do not design around): "optimizer discards same-day reviews".

## Goal
A complete, validated OFFLINE fitting pipeline: export a user's ReviewLog → fit FSRS-6 weights via
the canonical py-fsrs optimizer → holdout-evaluate fitted vs default → report. Shipped and proven
on SYNTHETIC data now (we have 0 real users); applying fitted weights to live scheduling is a
FUTURE wave gated on real volume + the measured holdout win. The ultimate validation is
WEIGHT RECOVERY: logs generated by OUR OWN engine under known weights must be recovered by the
fitter within tolerance — an external end-to-end oracle no self-consistent bug can fake.

## Design
- **Export** (`scripts/fsrs-fit/export-logs.ts`, tsx --conditions=react-server): per-user ReviewLog
  → the py-fsrs optimizer's expected CSV/schema (card_id, review_time, review_rating, engine tag
  filter: ONLY `fsrs6-bkt2` rows — grade semantics segmentation is exactly what the engine column
  exists for; document that pre-bkt2 rows are excluded and why).
- **Fitter** (`scripts/fsrs-fit/fit.py`, venv — house precedent wave19a: python3 -m venv,
  pip install 'fsrs==6.3.1' + the optimizer package; NETWORK ONLY in setup, the fit itself offline):
  wraps the canonical py-fsrs/fsrs-optimizer entry point; emits fitted 21-weight vector + train
  stats as JSON. Pin exact package versions in a requirements.txt.
- **Holdout eval** (`scripts/fsrs-fit/evaluate.py`): chronological split per user (train on first
  80% of reviews, evaluate log-loss/RMSE(bins) of predicted-R-at-review vs observed recall on the
  last 20%) for BOTH default and fitted weights. The prediction math (retrievability under a weight
  vector) is reimplemented in the python evaluator from the FSRS-6 formulas — cross-checked against
  our TS `retrievability` on frozen fixtures (the gen-19a/20 oracle discipline).
- **Synthetic-data validation** (the oracle of the wave):
  (a) GENERATOR (`scripts/fsrs-fit/gen-synthetic.ts`): simulate N users × M cards with the REAL TS
  engine under (i) DEFAULT weights and (ii) a PERTURBED vector (e.g. w8/w11/w20 shifted ±20%),
  recall sampled at true retrievability, realistic review cadence (due-driven + noise), 300–1500
  reviews/user — the exam-prep regime.
  (b) RECOVERY GATE: fitting on (ii)-generated logs must produce weights whose HOLDOUT log-loss is
  strictly better than the default vector's on that same synthetic population, and the fitted
  forgetting-curve decay (w20) must move TOWARD the true perturbed value (direction, not exact
  recovery — 300-review fits are noisy; the research's open question is precisely how noisy, so the
  report MEASURES recovery error vs review count at 200/500/1000/2000).
  (c) NULL GATE: fitting on (i)-generated logs must NOT beat defaults on holdout by more than a
  noise threshold (protects against an evaluator that always flatters the fit).
- **Report**: `docs/research/WEIGHTFIT-HARNESS-2026-07-14.md` — the recovery/null tables, the
  review-count curve (the survey's open question, answered on synthetic ground truth), and the
  data-gate recommendation for real users (min reviews, expected gain).

## Deliverables
1. venv + requirements + fit.py + evaluate.py, with a frozen-fixture cross-check of the python
   retrievability vs our TS on 6dp vectors (oracle-first task).
2. export-logs.ts (engine-tag filtered; integration test on a seeded fixture user).
3. gen-synthetic.ts (deterministic LCG; unit-pinned determinism).
4. The validation run (recovery + null gates) as an investigation task with committed verbatim
   outputs + FINDINGS.md (evaluator learnings apply).
5. Verify-wave: typecheck · unit · integration · build; NO live scheduling change anywhere —
   grep-gate that lib/fsrs/constants.ts FSRS_DEFAULT_WEIGHTS is byte-untouched and nothing imports
   fitted weights.

## Out of scope
Applying fitted weights live (future wave, data-gated); population-level refits; short-term
(same-day) modeling changes; anything in lib/fsrs.

## House rules
Standard. The py-fsrs optimizer is EXTERNAL canon — never reimplement the fit; our code only
generates, exports, evaluates, and cross-checks. PEP-668: system pip is blocked, use the venv.
Opus bulk. db:seed before integration in gates.
